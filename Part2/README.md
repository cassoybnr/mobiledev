For Part 2, I integrated the "sabotage" code where a TextView was set to null, which caused a NullPointerException because I tried to call .setText() on an object that hadn't been linked to the UI. This crash happens because the "Brain" (Java logic) must be connected to the "Face" (XML layout) using findViewById() during the onCreate() phase of the Activity Lifecycle. To handle the "mobile difference" where the OS destroys activities during screen rotation to save memory, I used onSaveInstanceState() to store my counter in a Bundle and onRestoreInstanceState() to retrieve it. This ensures the count doesn't reset to zero when the screen orientation changes, providing a seamless user experience despite the "destroy-and-recreate" cycle.
